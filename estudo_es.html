Sistemas de Informação

Definição de Sistema

Sommerville: "Um sistema é uma coleção de componentes inter-relacionados que trabalham juntos para atingir um objetivo comum."

Sistema Físico (concreto)

Existe fisicamente — ex: computador, carro, avião.

Sistema Abstrato

Sem existência física — ex: modelo matemático, software.

Composição de um Sistema de Informação

ENTRADAS → PROCESSAMENTO → SAÍDAS
↑_______________ FEEDBACK _______________↑

Ecossistema de um SI — 5 Elementos

🖥 Hardware

Componentes físicos: CPU, memória, armazenamento, periféricos.

💾 Software

Programas e sistemas operacionais.

📊 Dados

Matéria-prima do SI; transformados em informação.

🌐 Conectividade

Redes que interligam os componentes.

👤 Pessoas

Usuários, analistas, administradores.

  • Tático: gerência média, controle e análise (ex: MIS)
  • Operacional: rotina do dia-a-dia (ex: ERP, CRM, SPT)

Elementos de um Sistema

Objetos

Partes e componentes do sistema.

Atividades

Funções realizadas pelos objetos.

Relações

Ligações entre objetos e atividades.

Fronteira (Boundary)

O que está dentro ou fora do escopo.

Sistema Fechado não interage com o ambiente. Sistema Aberto troca informações com o exterior — SI são sistemas abertos.

Análise e Desenvolvimento de Sistemas

O que é ADS?

Área que estuda como criar software de qualidade de forma sistemática, controlada e eficiente.

Crise do Software

Surgiu no final da década de 1960. Problemas: projetos atrasados, acima do orçamento, sem qualidade e não atendiam requisitos. Cunhada na Conferência da OTAN (1968).

Mitos do Software

CategoriaMitoRealidade
Administrativos "Basta adicionar mais programadores para recuperar atraso" Mais pessoas = mais comunicação = mais atraso (Lei de Brooks)
Cliente "Já sei o que quero; especificação detalhada depois" Requisitos mal definidos são a principal causa de falha
Profissional "Quando o programa funcionar, o trabalho está feito" Manutenção consome 60-80% do custo total de um software
Profissional "Posso usar software já pronto e adaptá-lo" Reuso exige planejamento; adaptação pode custar mais

Análise vs Síntese

Análise

Divisão do problema em partes menores para compreensão. Ex: decompor o sistema em módulos.

Síntese

Integração das partes em uma solução coesa. Ex: integrar módulos na solução final.

Meios de Desenvolvimento

Método

Abordagem sistemática — como fazer. Ex: Scrum, Waterfall.

Ferramenta

Suporte automatizado — com o que fazer. Ex: IDE, ferramentas CASE.

Procedimento

Sequência de passos — o que fazer quando. Ex: processo de revisão de código.

Crise do Software → final dos anos 60. Lei de Brooks: "adicionar mão de obra a um projeto atrasado o atrasa ainda mais."

Modelo Cascata e Processos

O que é Processo de Software?

Conjunto de atividades, métodos e práticas que guiam o desenvolvimento de software. Define quem faz o quê, quando e como.

Inclui: produtos (artefatos), papéis e restrições de pré/pós-condições.

Modelo Caótico — "Go Horse" ⚠

NÃO É um modelo válido. Também chamado de "Cowboy Coding". Desenvolvimento sem planejamento, documentação ou processos. Altamente disfuncional.

Postulatos do Go Horse: "Primeiro faço, depois penso", "Se funciona, não mexa", etc.

Modelo Cascata (Waterfall)

Criado por Winston Royce (1970). Modelo linear e sequencial — cada fase deve ser concluída antes da próxima iniciar.

1. Definição de Requisitos
↓
2. Projeto de Sistema
↓
3. Implementação
↓
4. Testes (Integração e Verificação)
↓
5. Operação e Manutenção

✅ Vantagens

  • Simples de entender e gerenciar
  • Documentação clara em cada fase
  • Bom para requisitos bem definidos e estáveis
  • Controle de qualidade por fases

❌ Desvantagens

  • Mudanças de requisitos são caras
  • Cliente só vê o produto no final
  • Difícil voltar fases anteriores
  • Não reflete o desenvolvimento real

Variações do Cascata

Modelo V

Cada fase de desenvolvimento tem uma fase de teste correspondente. Verificação e validação são planejadas junto com o desenvolvimento.

Codificação ↔ Teste Unitário | Design ↔ Teste de Integração | Análise ↔ Teste de Aceitação

Prototipação

Criação de protótipos para validar requisitos antes do desenvolvimento completo. Reduz riscos de mal-entendidos. Protótipo pode ser descartável ou evolutivo.

O Royce criou o Cascata em 1970. Modelo V é extensão do Cascata com foco em testes. Go Horse NÃO é um modelo de processo válido.

Modelo Iterativo, Incremental e RUP

Iterativo vs Incremental

Iterativo

Refina e melhora o produto através de ciclos repetidos. O sistema completo é desenvolvido em cada iteração, melhorando progressivamente. "Versões cada vez mais completas."

Incremental

Entrega partes funcionais do sistema separadamente. Cada incremento adiciona novas funcionalidades. "Partes do sistema entregues progressivamente."

Na prática, os modelos modernos combinam ambas as abordagens: cada ciclo (iteração) entrega um incremento funcional do sistema.

Modelo Espiral (Boehm, 1988)

Combina prototipação e cascata. A cada espiral: Planejamento → Análise de Riscos → Engenharia → Avaliação. Foco em gestão de riscos. Bom para projetos grandes e complexos.

RUP / UP — Processo Unificado

Rational Unified Process — criado por Kruchten (1995), baseado no UML. Processo iterativo e incremental, orientado a casos de uso, centrado na arquitetura.

4 Fases do RUP

FaseObjetivo PrincipalMarco
Concepção (Inception) Escopo, viabilidade, casos de negócio, principais riscos Objetivo do ciclo de vida
Elaboração Análise do problema, arquitetura base, eliminação dos principais riscos Arquitetura do ciclo de vida
Construção Desenvolvimento iterativo, implementação de todos os casos de uso Capacidade operacional inicial
Transição Entrega ao usuário final, testes beta, treinamento, correções finais Lançamento do produto

Disciplinas do RUP

Engenharia

  • Modelagem de negócio
  • Requisitos
  • Análise e Design
  • Implementação
  • Teste
  • Implantação

Suporte

  • Gerência de configuração
  • Gerência de projeto
  • Ambiente

Boas Práticas do RUP

  • Desenvolver software iterativamente
  • Gerenciar requisitos continuamente
  • Usar arquitetura baseada em componentes
  • Modelar software visualmente (UML)
  • Verificar a qualidade continuamente
  • Controlar mudanças no software
RUP = Kruchten 1995, 4 fases (Concepção → Elaboração → Construção → Transição). Espiral = Boehm 1988, foco em riscos.

Modelos Ágeis e Scrum

Manifesto Ágil (2001)

Publicado em fevereiro de 2001 por 17 profissionais em Snowbird, Utah. Base filosófica do desenvolvimento ágil.

4 Valores

Indivíduos e interações sobre processos e ferramentas

Software funcionando sobre documentação abrangente

Colaboração com o cliente sobre negociação de contratos

Responder a mudanças sobre seguir um plano

* Os itens à direita têm valor, mas os à esquerda são mais valorizados.

12 Princípios (resumidos)

  1. Satisfazer o cliente com entregas contínuas e antecipadas de software
  2. Aceitar mudanças de requisitos, mesmo tarde no desenvolvimento
  3. Entregar software funcionando frequentemente (semanas, não meses)
  4. Clientes e desenvolvedores devem trabalhar juntos diariamente
  5. Construir projetos em torno de indivíduos motivados
  6. Conversa face a face é a comunicação mais eficiente
  7. Software funcionando é a medida de progresso
  8. Processos ágeis promovem desenvolvimento sustentável
  9. Atenção contínua à excelência técnica e bom design
  10. Simplicidade — maximizar o trabalho não realizado
  11. Times auto-organizáveis geram melhores arquiteturas
  12. Reflexão regular sobre como se tornar mais eficaz

Requisitos de Software

Definições

Sommerville: "Descrições do que o sistema deve fazer — os serviços que oferece e as restrições a seu funcionamento."

Pleeger: "Uma condição ou capacidade que o sistema deve ter para satisfazer um contrato, padrão ou especificação."

Classificação — Pirâmide

Requisito de Negócio
Requisito de Usuário
Requisito de Sistema
  • Negócio: objetivos de alto nível da organização. Ex: "Aumentar vendas online em 20%"
  • Usuário: o que o usuário espera fazer com o sistema. Ex: "Quero pesquisar produtos"
  • Sistema: especificação detalhada do que o software deve fazer.

Tipos de Requisitos

Funcionais (RF)

O que o sistema faz. Descrevem funcionalidades, serviços, comportamentos.

Ex: "O sistema deve permitir login com email e senha."

Não-Funcionais (RNF)

Restrições sobre o como. Qualidade, desempenho, segurança, usabilidade.

Ex: "O sistema deve responder em menos de 2 segundos."

Subtipos de RNF

Produto

Desempenho, confiabilidade, portabilidade, usabilidade.

Organizacionais

Padrões, processos de desenvolvimento usados.

Externos

Legais, regulatórios, éticos, interoperabilidade.

Técnicas de Levantamento

  • Entrevistas: estruturadas ou não. Direta com stakeholders.
  • Questionários: para grande número de usuários dispersos.
  • Observação: analista observa o ambiente de trabalho.
  • Workshops/JAD: sessões colaborativas com stakeholders.
  • Análise de documentos: estudar documentos existentes.
  • Prototipação: criar modelo parcial para validar entendimento.
  • Brainstorming: geração livre de ideias.
  • Cenários / User Stories: descrição narrativa de como o sistema será usado.

Características de um Bom Requisito

  • ✅ Completo: contém todas as informações necessárias
  • ✅ Correto: representa com precisão a necessidade
  • ✅ Consistente: não conflita com outros requisitos
  • ✅ Não ambíguo: apenas uma interpretação possível
  • ✅ Verificável: pode ser testado
  • ✅ Rastreável: origem identificada
  • ✅ Modificável: pode ser alterado com facilidade
  • ✅ Priorizável: tem grau de importância

Problemas Comuns nos Requisitos

Escopo

Objetivos mal definidos ou escopo inflado desnecessariamente.

Entendimento

Cliente não sabe comunicar o que precisa; analista mal interpreta.

Volatilidade

Requisitos mudam com frequência durante o desenvolvimento.

RF = o que o sistema FAZ. RNF = como o sistema SE COMPORTA. Pirâmide: Negócio → Usuário → Sistema (do topo à base).

Desenho / Modelagem de Software

Definição

Processo de traduzir os requisitos (o quê) em uma representação de como o sistema será construído. É a ponte entre análise e implementação.

Envolve tomada de decisões sobre estrutura, algoritmos e interfaces.

Visões do Desenho

Visão Conceitual

O que o sistema faz do ponto de vista do problema. Independente de tecnologia. Ex: Diagrama de Casos de Uso, modelos de domínio.

Visão Técnica

Como o sistema será construído. Focada em solução. Ex: Diagramas de Classe, Sequência, Arquitetura.

Conceitos Fundamentais

ConceitoDescrição
AbstraçãoFoco nos aspectos essenciais, ignorando detalhes não relevantes para o nível atual.
RefinamentoProcesso inverso da abstração — elaborar detalhes progressivamente (top-down).
ModularizaçãoDivisão do sistema em partes independentes (módulos) com interfaces definidas.
Hierarquia de controleOrganização dos módulos em estrutura hierárquica — quem controla quem.
Ocultamento de informaçãoEncapsulamento — detalhes internos não ficam expostos (information hiding).
Independência funcionalMódulos com função única e bem definida, com mínima dependência entre si.

Modelos Técnicos

Modelo de Arquitetura

Estrutura macro do sistema. Define componentes principais e como se comunicam. Ex: Diagrama de Componentes, Implantação.

Modelo de Procedimentos

Como as funções são executadas internamente. Sequência de ações. Ex: Diagrama de Sequência, Atividades, Fluxogramas.

Modelo de Dados

Estrutura e organização dos dados. Ex: Diagrama ER, Diagrama de Classes (atributos).

Arquitetura em Camadas

Apresentação (UI) — Interface com usuário
Negócio (Business Logic) — Regras e processamento
Dados (Persistence) — Acesso ao banco de dados
Abstração = visão de alto nível (esconde detalhes). Refinamento = detalhar progressivamente. Modularização = dividir para conquistar.

Arquitetura de Software

Definições

Bass et al.: "A arquitetura de software de um sistema é a estrutura (ou estruturas) do sistema, que compreende os elementos de software, as propriedades externamente visíveis desses elementos e os relacionamentos entre eles."

IEEE 1471: Organização fundamental de um sistema, incorporada em seus componentes, nos relacionamentos entre si e com o ambiente, e os princípios que guiam seu design e evolução.

Conceitos: Serviço, Componente, Módulo

ConceitoDefiniçãoExemplo
MóduloUnidade de implementação com interface definida (código)Classe Java, arquivo Python
ComponenteUnidade de software implantável e substituível com interface contratualJAR, DLL, container Docker
ServiçoComponente acessível pela rede com contrato bem definido (API)REST API, microserviço

Influência nos Requisitos Não-Funcionais

A arquitetura afeta diretamente:

🔒 Segurança

Camadas de autorização, isolamento de componentes sensíveis.

⚡ Desempenho

Cache, balanceamento de carga, comunicação assíncrona.

📈 Escalabilidade

Horizontal (mais servidores) ou vertical (mais recursos).

🔧 Manutenibilidade

Separação de responsabilidades facilita mudanças.

✅ Disponibilidade

Redundância, failover, replicação.

🔄 Interoperabilidade

Protocolos e contratos padronizados.

Padrões / Estilos Arquiteturais

Cliente-Servidor

Clientes requisitam serviços a servidores. Separação clara entre quem consome e quem provê. Ex: navegador/servidor web.

MVC — Model-View-Controller

Model: dados e regras de negócio. View: apresentação (UI). Controller: intermediário que processa entrada do usuário e atualiza Model/View. Separa responsabilidades.

Multicamada (N-Tier)

Divisão em camadas: Apresentação, Lógica de Negócio, Dados. Cada camada só se comunica com a adjacente. Variante do MVC com foco em implantação física.

SOA — Arquitetura Orientada a Serviços

Sistema composto por serviços independentes que se comunicam por interfaces padronizadas (SOAP/REST). Altamente reutilizável e integrável. Base para Microserviços.

Boas Práticas Arquiteturais

  • Separação de responsabilidades (Separation of Concerns)
  • Baixo acoplamento entre componentes
  • Alta coesão dentro dos componentes
  • Documentar decisões arquiteturais (ADR)
  • Considerar requisitos não-funcionais desde o início
  • Preferir composição a herança excessiva
MVC = separação UI/Lógica/Dados. SOA = serviços com contratos padronizados. Arquitetura impacta TODOS os RNF (desempenho, segurança, manutenibilidade...).

Construção / Implementação

O que é Construção?

Fase onde o design é convertido em código executável. Envolve programação, testes unitários, integração e depuração. É a fase mais longa e cara do desenvolvimento.

Algoritmos

Fluxograma

Representação gráfica do algoritmo usando símbolos padronizados:

  • ⬭ Oval: Início/Fim
  • ▭ Retângulo: Processo/Instrução
  • ◇ Losango: Decisão (if/else)
  • ▱ Paralelogramo: Entrada/Saída
  • → Setas: Fluxo de controle

Pseudocódigo

Descrição textual informal do algoritmo, usando linguagem próxima do humano mas com estrutura de código. Independente de linguagem de programação.

Ocultamento de detalhes internos. Acesso via interface pública.

Abstração

Modelar apenas o que é relevante para o contexto.

Nível de Aplicação

Reutilizar sistemas inteiros. Ex: usar um ERP existente em vez de desenvolver do zero.

Nível de Componente

Reutilizar subsistemas ou módulos. Ex: bibliotecas, frameworks, APIs.

Nível de Objeto/Função

Reutilizar classes, funções isoladas. Menor granularidade.

Gerência de Configuração

  • Controle de versão (Git, SVN)
  • Gerência de releases
  • Controle de mudanças
  • Auditoria e status
Métodos de programação: Lógica → Dados → Estruturada → OO (cronológico). OO = 4 pilares: Encapsulamento, Herança, Polimorfismo, Abstração.

Qualidade de Software

Meta Principal: Baixo Acoplamento + Alta Coesão

⬇ Acoplamento — quanto menor, melhor

Grau de dependência entre módulos. Alto acoplamento = mudança em um módulo afeta muitos outros. Dificulta manutenção e testes.

⬆ Coesão — quanto maior, melhor

Grau em que os elementos de um módulo pertencem juntos. Alta coesão = módulo faz uma coisa bem. Facilita manutenção e reutilização.

Tipos de Acoplamento (do pior para o melhor)

TipoDescriçãoQualidade
ConteúdoMódulo acessa diretamente o interior de outroPéssimo
ComumCompartilham variáveis globaisMuito ruim
ExternoDependem de formatos externos (arquivo, protocolo)Ruim
ControleUm passa flags de controle para outroMédio
Stamp (carimbo)Passam estruturas de dados (parte não usada)Aceitável
DadosPassam apenas dados simples necessários (parâmetros)Ótimo

Tipos de Coesão (da pior para a melhor)

TipoDescriçãoQualidade
CoincidentalElementos agrupados por acaso, sem relaçãoPéssima
LógicaAtividades do mesmo tipo geral (ex: todas as I/O)Ruim
TemporalAtividades executadas no mesmo momento (ex: inicialização)Fraca
ProceduralAtividades em sequência de execuçãoMédia
ComunicacionalAtividades que operam nos mesmos dadosBoa
SequencialSaída de uma atividade é entrada da próximaMuito boa
FuncionalTodos os elementos contribuem para uma única função bem definidaIdeal

Tratamento de Exceções

  • Exceções devem ser tratadas no nível adequado (onde há contexto para tratar)
  • Não usar exceções para controle de fluxo normal
  • Logar informações suficientes para diagnóstico
  • Nunca capturar exceção genérica e ignorar silenciosamente
  • Restaurar estado consistente após exceção

Defeito vs Falha

Defeito (Fault/Bug)

Problema no código-fonte. Pode existir sem jamais se manifestar se o trecho não for executado.

Falha (Failure)

Manifestação observável de um defeito em tempo de execução. O sistema não se comporta conforme esperado.

Erro → causa humana → gera Defeito no código → ativado em execução → produz Falha visível ao usuário.

Acoplamento: Conteúdo (pior) → Dados (melhor). Coesão: Coincidental (pior) → Funcional (melhor). Meta: baixo acoplamento + alta coesão.

Implantação e Manutenção

Implantação (Deployment)

Fase de entrega do software ao ambiente de produção e aos usuários finais. Inclui instalação, configuração, testes finais, treinamento e documentação.

Treinamento

Treinamento do Usuário Final

Foca nas funcionalidades do dia-a-dia. Como usar as telas, cadastros, relatórios. Linguagem acessível, exemplos práticos do contexto do usuário.

Treinamento do Operador

Foca na operação técnica do sistema. Backup, monitoramento, restart de serviços, gestão de usuários. Linguagem mais técnica.

Manutenção de Software

IEEE: "Modificação de um produto de software após a entrega para corrigir falhas, melhorar desempenho ou outros atributos, ou adaptar o produto a um ambiente modificado."

A manutenção consome 60 a 80% do custo total de um software ao longo de sua vida útil.

4 Tipos de Manutenção

TipoGatilhoExemplo
Corretiva Correção de defeitos/falhas já existentes Corrigir bug que causa crash no login
Adaptativa Mudanças no ambiente externo Migrar para novo SO, nova versão do BD, nova legislação
Perfectiva (Evolutiva) Novas funcionalidades ou melhorias Adicionar módulo de relatórios, melhorar performance
Preventiva Evitar problemas futuros Refatorar código antes que vire problema, melhorar documentação

Tipos de Sistema — Lehman

Sistema S (Specified)

Definido completamente por uma especificação formal. Ex: algoritmo matemático. Não muda.

Sistema P (Problem)

Resolução de um problema do mundo real. Pode ser refinado mas o problema é estável.

Sistema E (Embedded)

Embutido no mundo real; evolui conforme o ambiente muda. Ex: sistemas de negócio. Sujeito às Leis de Lehman.

Manutenibilidade

Facilidade com que um software pode ser modificado. Fatores:

  • Compreensibilidade: quão fácil é entender o código
  • Testabilidade: quão fácil é testar mudanças
  • Modificabilidade: quão fácil é implementar mudanças

Rejuvenescimento de Software

Reestruturação

Reorganizar o código sem mudar funcionalidade. Ex: refatoração, eliminar código duplicado.

Redocumentação
